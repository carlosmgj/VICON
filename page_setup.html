PYTHON

Warning
Realizado downgrade de Python de 3.14 a 3.13 porque el primero no tiene todavía soporte para la librería "cocotb".
Instalado adicionalmente Python 3.11.9 (sin añadir al PATH del sistema) porque VUnit y pyGHDL tienen soporte oficial hasta Python 3.11 o 3.12. Ambas versiones conviven usando el launcher py de Windows (py -3.13 / py -3.11).

Durante la instalación de Python 3.11: NO marcar "Add python.exe to PATH". Sí marcar "Install launcher for all users".

Verificar que ambas versiones conviven correctamente:

Necesario para que Doxygen pueda invocar PlantUML (que está escrito en Java).

Note
Durante la instalación marcar: ✅ Add to PATH y ✅ Set JAVA_HOME.

Verificar:

java -version # openjdk version "21.x.x"

GHDL

GHDL es el compilador/simulador VHDL open source. En este proyecto se usa únicamente como parser para que pyGHDL pueda analizar el AST de los ficheros VHDL y generar documentación automática. QuestaSim sigue siendo el simulador principal — no hay conflicto entre ambos.

Binario GHDL

Descomprimir en C:\ghdl y añadir C:\ghdl\bin al PATH del sistema.

Verificar:

ghdl --version # GHDL 6.0.0

pyGHDL (bindings Python)

La distribución Windows de GHDL no incluye los bindings Python. Hay que clonar el source de la versión exacta que coincide con el binario instalado (6.0.0-rc2):

git clone --depth=1 --branch v6.0.0-rc2 https://github.com/ghdl/ghdl.git C:\tools\ghdl-src-6
setx PYTHONPATH "C:\tools\ghdl-src-6"
Warning
Abrir una terminal nueva después de setx para que el PYTHONPATH tenga efecto.

Instalar dependencias de pyGHDL con Python 3.11:

py -3.11 -m pip install pyTooling pyVHDLModel

Verificar que pyGHDL funciona correctamente:

py -3.11 -c "from pyGHDL.dom.NonStandard import Design, Document; print('OK')"

VUNIT

VUnit es el framework de verificación VHDL. En este proyecto se usa para:

  • Definir el orden de compilación del proyecto (necesario para pyGHDL)
  • Lanzar simulaciones con QuestaSim
py -3.11 -m pip install vunit-hdl

Verificar:

py -3.11 -m pip show vunit-hdl # Version: 4.x.x

WaveDrom (offline)

WaveDrom permite incluir diagramas de timing en la documentación. Se carga localmente para que la documentación funcione sin conexión a internet.

Descargar los siguientes ficheros y colocarlos en doxygen/style/extra_js/:

  • https://cdnjs.cloudflare.com/ajax/libs/wavedrom/3.1.0/wavedrom.min.js
  • https://cdnjs.cloudflare.com/ajax/libs/wavedrom/3.1.0/skins/default.js

En el Doxyfile:

HTML_EXTRA_FILES = style/extra_js/wavedrom.min.js \
style/extra_js/default.js

En header.html, añadir antes de </head>:

<script src="$relpath^wavedrom.min.js" type="text/javascript"></script>
<script src="$relpath^default.js" type="text/javascript"></script>

Y el tag <body>:

<body onload="WaveDrom.ProcessAll()">

SCRIPT DE DOCUMENTACIÓN VHDL

El script scripts/vhdl_dox_filter.py es el núcleo del sistema de documentación. Se configura en el Doxyfile como INPUT_FILTER y Doxygen lo invoca automáticamente en cada fichero .vhd.

Qué hace

El script usa pyGHDL para parsear el AST real del VHDL (no regex) y genera automáticamente, sin tocar el código fuente:

Elemento Fuente en el AST
Tabla de genéricos entity.GenericItems
Tabla de puertos entity.PortItems
Diagrama de entidad Puertos + señales internas
Tabla de señales internas arch.DeclaredItems (Signal)
Tabla de constantes arch.DeclaredItems (Constant)
Diagrama FSM arch.Statements → ProcessStatement → CaseStatement

Convención de comentarios en el .vhd

Solo necesitamos escribir:

  1. \brief en el comentario de la entidad (título de la página)
  2. Comentarios --! inline en cada puerto (descripción)
  3. Bloque \htmlonly con WaveDrom si se necesita timing diagram (opcional)

El script genera todo lo demás automáticamente.

Ejemplo mínimo: 

--! \brief Módulo de control con FSM de 3 estados
--! \htmlonly
--! <script type="WaveDrom">
--! { "signal": [
--!   { "name": "clk", "wave": "P......." },
--!   { "name": "rst", "wave": "10......" }
--! ]}
--! </script>
--! \endhtmlonly
entity mi_modulo is
    port (
        clk : in  std_logic;  --! Reloj del sistema (activo en flanco de subida)
        rst : in  std_logic;  --! Reset síncrono activo alto
        y   : out std_logic   --! Salida del módulo
    );
end entity;

TEROSHDL

Extensión de VS Code para diseño digital. Proporciona visualización de entidades, diagramas de estados y timing diagrams de forma interactiva mientras se edita.

Note
TerosHDL no es compatible directamente con los comandos de Doxygen en los comentarios. Se usa como herramienta de visualización durante el desarrollo, no para generar la documentación final del proyecto.

Instalar desde VS Code: buscar "TerosHDL" en las extensiones.

TEROSHDL CLI

Para usar la CLI de TerosHDL (colibri) para generar artefactos que luego se inserten en Doxygen, se necesita Node.js y npm.

Warning
La versión 24 de Node.js da problemas con TerosHDL. Instalar la versión 18.

Instalar TerosHDL CLI:

npm install -g teroshdl

Parche necesario para que funcione en entornos Node.js sin entorno de navegador:

$treeSitter = "$env:APPDATA\npm\node_modules\teroshdl\node_modules\web-tree-sitter\tree-sitter.js"
$content = Get-Content $treeSitter -Raw
# Parche 1: inyectar wasmBinary al inicio
$patch = "var Module=Module||{};Module.wasmBinary=require('fs').readFileSync(require('path').join(__dirname,'tree-sitter.wasm'));`n"
$content = $patch + $content
# Parche 2: forzar entorno Node.js
$content = $content -replace 'a=!1,_=!1;a=\"object\"==typeof window,_=\"function\"==typeof importScripts,n=\"object\"==typeof process&&\"object\"==typeof process\.versions&&\"string\"==typeof process\.versions\.node', 'a=!1,_=!1;a=false,_=false,n=true'
# Parche 3: Uint8Array para carga de .wasm de lenguaje
$content = $content -replace 'const n=require\‍(\"fs\"\‍);t=Promise\.resolve\‍(n\.readFileSync\‍(e\‍)\‍)', 'const n=require("fs");t=Promise.resolve(new Uint8Array(n.readFileSync(e)))'
Set-Content $treeSitter $content
Write-Host "Parches aplicados correctamente"

Para ver como quedaría lanzar el comando mostrado en TEROSHDL

YOSYS

Todo
Ver cómo generar diagramas con Yosis e incluirlos en la doc. ¿Merece la Pena?

Ocupa mucho todo el toolchain. Yosys tiene compatibilidad nativa con Verilog pero no con VHDL. Habría que instalar un plugin para yosys-vhdl.

El problema en Windows es que ghdl-yosys-plugin no tiene binarios precompilados para Windows — hay que compilarlo desde fuente, lo que requiere un entorno MINGW/MSYS2 y puede ser tedioso. La opción más práctica en Windows es el oss-cad-suite.

En mi opinión No merece la pena si es únicamente para diagramas de esquemático. Pero si lo queremos lo podemos explorar
